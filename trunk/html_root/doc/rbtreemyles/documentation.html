Red Black Trees

Like their cousin, the binary search tree, Red Black trees continue to have a binary search property where all nodes larger than the current node are to the right and all nodes smaller are to the left. The difference is, however, that Red Black trees incorporate a self balancing technique when adding and deleting nodes in order to keep the tree fairly well balanced.

The properties of a Red Black tree are as follows:

  • The root is always black,
  • Every external node is black,
  • Both children of a red node must be black, and
  • all external nodes will have the same black depth (number of black ancestors minus one).

Adhering to these properties causes the insert and delete methods to modify the tree whenever a rule is broken after a node is inserted or deleted.

With insert there are two basic situations that it must handle:

  1. The uncle of the added node is black, which will cause a restructuring of the nodes.
  2. The uncle of the added node is red, which causes a recoloring of the structure.

With delete we must examine the location of where the deleted node got its new value from (what node took the deleted nodes place). And with this there are three cases to consider:

  1. Original sibling of node that took deleted nodes place is black and has a red child, which causes a need for restructuring of the sibling its parent and its with the final root of the adjusting subtree being black.
  2. The original sibling of the node that replaced the deleted node is black and has no red children, this causes a need for recoloring only, however, the tree will have to be traversed up to make sure recoloring didn't cause other "double black" problems.
  3. The original sibling of node that replaced the deleted node is red, this causes a need for restructuring of the sibling its child and its parent with the final root of the adjusting subtree being red.

With a Red Black tree the operations on it end up having the following times:
Search: O(log n)
insert: O(log n)
delete: O(log n)
